web性能优化

发表于 2021-06-09

性能是用户体验的一个重要方面。网页的加载速度和响应用户输入的速度会显著影响用户对您网站的看法，并且还会决定用户是继续浏览您的网站还是放弃您的网站。

为什么性能优化如此重要

  • 性能对用户的影响
  • 留存率
  • 转化率
  • 品牌形象
  • 用户满意度

从输入 URL 到页面加载完成 中间发生了什么？

  1. 处理URL

    • 浏览器判断是搜索词还是合法URL
    • 补全协议，默认HTTPS
    • 检查 HSTS，（HSTS是HTTP严格传输安全（HTTP Strict Transport Security）的缩写）

  2. 本地强缓存判断

    • 先查 Memory Cache (内存缓存)
    • 再查 Disk Cache (磁盘缓存)
    • 依据：Cache-control: max-age=xxx(设置资源在本地缓存的有效时间[秒]，‌优先级最高‌‌), Expires (HTTP/1.0 绝对过期时间，若与 Cache-Control 同时存在，‌以 max-age 为准‌‌)
    • 命中 -> 使用本地资源不走网络，直接进入解析渲染流程
    • 未命中 -> 继续往下走网络请求

  3. DNS解析(域名 -> IP地址)

    • 浏览器缓存‌：检查近期解析结果
    • 操作系统缓存‌：查询 hosts 文件
    • 本地 DNS 服务器（LDNS）‌：由 ISP(国内三大运营商之流) 提供，通常能完成 80% 的解析
    • 根域名服务器 → 顶级域（如 .com）→ 权威域名服务器‌：逐级查询直至获得 IP 地址。

      此过程遵循“缓存优先、逐层递归”原则，耗时通常在几毫秒到数百毫秒之间 ‌

  4. 建立TCP连接

    • 浏览器使用解析得到的 IP 地址与服务器建立 TCP 连接。
    • 通过 ‌三次握手‌ 确认连接：
      1. 客户端发送 SYN 包；
      2. 服务器回复 SYN+ACK 包；
      3. 客户端发送 ACK 包，连接建立 ‌

        若使用 HTTPS，还需额外进行 ‌TLS 握手‌（协商加密算法、交换密钥等），以建立安全通道

  5. 发送 http 请求， 带上协商缓存头(ETag + If-None-Match)

    • 浏览器构造 HTTP 请求报文，包含：
      • ‌请求行‌：方法（如 GET）、路径、协议版本；
      • 请求头‌：如 Host、User-Agent、Accept-Encoding、Cookie 等
      • 请求体‌（仅 POST/PUT 等方法有）。

  6. 服务器处理并返回响应

    • 服务器接收请求，由 Web 服务（如 Nginx、Apache）处理。
    • 根据请求路径调用对应程序（如 PHP、Node.js、java），生成响应。
    • 304 -> 使用本地缓存
    • 200 -> 返回新 HTML
    • 响应包含：
      • 状态码‌（如 200 OK、301 Moved Permanently、404 Not Found）；
      • 响应头‌：如 Content-Type、Cache-Control、Set-Cookie；
      • ‌响应体‌：通常是 HTML 文档

  7. 构建渲染树并布局绘制

    • 合并 DOM 和 CSSOM，生成 ‌渲染树（Render Tree）‌，仅包含可见节点。
    • 计算每个节点的几何位置和尺寸（‌布局/Layout‌）。
    • 将渲染树转换为像素（‌绘制/Paint‌），并分层处理（如定位元素、透明层等）‌

  8. 加载子资源与交互绑定

    • 解析 HTML 中的 <script>、<img>、<link> 等标签，‌递归发起请求‌加载 CSS、JS、图片等资源。
    • 执行 JavaScript（可能修改 DOM/CSSOM，触发重排重绘）。
    • 页面完全加载后，触发 DOMContentLoaded 和 load 事件，用户可与页面交互 ‌

只有深刻理解了一个网页从 URL 到页面展示的整个过程中发生了什么，我们才能知道做优化的时候，可以从哪些方面入手

性能指标

  • FCP: 首次绘制内容时间（First Contentful Paint）, 时间越短意味着页面白屏时间越短。低于1.8s代表良好，1.8s到3s之间代表待改善，3s以上代表差。
  • LCP: 最大绘制内容时间（Largest Contentful Paint）。低于2.5s代表良好，2.5s到4s之间代表待改善，4s以上代表差。
  • FID：首次输入延迟，评估用户首次与页面交互时的响应速度。良好的FID应小于100毫秒。
  • CLS：累积布局偏移（Cumulative Layout Shift），衡量页面上元素位置发生变化的频率和程度。低于0.1s代表良好，0.1s到0.25s之间代表待改善，0.25s以上代表差。
  • TTFB：首个字节返回时间（Time To First Byte），时间越短意味着请求响应越快。低于800ms代表良好，800ms到1800ms之间代表待改善，1800ms以上代表差。
  • FID: (First Input Delay),页面首次加载后用户第一次交互‌（如点击、按键）到浏览器实际响应的时间。
  • INP：下次渲染可交互时间（Interaction Next Paint），整个页面会话期间所有可交互操作‌（点击、触摸、键盘输入）的延迟。并取最慢的一次（或接近最慢的）作为最终得分，反映整体交互体验‌
    ，低于200ms代表良好，200ms到500ms之间代表待改善，500ms以上代表差。
  • …
  • 核心指标：LCP、CLS、INP

优化手段

web性能优化有很多手段，总体分为两大类

  1. 网络层面的优化：更快的网络请求、更少的网络请求
  2. 渲染层面的优化：更快的渲染页面、更少的渲染节点

从刚刚讨论的 【URL 到页面加载完成】的过程中我们可以针对每个流程做出相应的优化

浏览器缓存

一、 优先用：强缓存（完全不发请求，最快）

  1. 静态资源设置长 Cache-Control

    • HTML 一般不做强缓存（或短时间）
    • JS / CSS / 图片 / 字体：设置 max-age=31536000（1 年）
    • 浏览器直接从 Memory / Disk Cache 读取，不发网络请求

  2. 资源文件名带哈希（版本化）

DNS 优化

  • 减少 DNS 查询次数 提前解析关键域名 提升页面加载速度
  • DNS 预取 (DNS Prefetching)
  • dns-prefetch仅限于DNS解析，而preconnect不仅包含DNS解析还包括TCP连接建立以及可能的TLS握手
  • 由于preconnect涉及更多的网络操作，它比dns-prefetch消耗更多带宽和系统资源。因此应该谨慎使用，并确保只对确实需要尽快加载的重要资源进行预连接。
  • 如果只是想加快偶尔使用的第三方服务或者不确定是否会用到的资源，则选择dns-prefetch；若明确即将频繁使用某域名下的资源，则应考虑使用preconnect以获得更好的性能提升

Http 请求优化

  • 使用 http2/http3 协议，减少队头阻塞
  • 合并 js、css、图片等文件，合并成一个文件，（通常是同一个页面内的小图标合并成雪碧图，js/css 在框架层面已经处理了同一个路由内的代码会单独打包）
  • 接口合并。前端交互，经常需要请求多个并行或串行接口，此时可以通过接口合并方式，提高接口访问速度
  • 能使用css的尽量不使用js，能使用js的尽量不用三方插件，避免三方插件大量的库。
  • 减少cookie传输
  • 使用CDN提供静态文件
  • 启用 Gzip/Brotli/Zstandard 压缩
  • 分域存放资源
  • 减少页面重定向
  • 压缩 html、css、js 等文件
  • 在 js 之前引用 css（js 执行的时候会进入阻塞，如果放入 js 之后加载，会等待 js 执行完成之后才能加载 css，渲染页面）
  • 非阻塞 js (defer 和 async)
  • 矢量图替代位图、使用现代化的图片格式方案（webp/avif）

总结

性能优化的本质是，让用户能更快的看到更流畅的网页，了解网页请求与渲染的流程与原理可以帮助我们更好的进行性能优化
我们在做性能优化的时候，要从加载性能、运行性能、体验性能、监控体系多个维度系统性推进，做到可落地、可量化、可复盘。

加载性能优化（最基础也最能体现工程能力）

  1. 网络层面

    • 启用 HTTP/2、HTTP/3，减少队头阻塞
    • 配置合理缓存策略：强缓存 + 协商缓存
    • 资源压缩：Gzip/Brotli、图片 WebP/AVIF、字体子集化

  2. 资源体积与加载

    • 构建优化：Tree Shaking、代码压缩、去除 console、dead code
    • 路由懒加载、组件异步加载
    • 第三方依赖优化：替换轻量库、CDN 引入、动态 import

  3. 首屏优化

    • 关键 CSS 内联、非关键 CSS 异步加载
    • 图片懒加载、占位骨架屏
    • 预加载、预连接、预 fetch（按场景）

运行时性能优化（高级前端核心）

  1. JS 执行优化

    • 避免长任务，拆分大块计算
    • 防抖节流处理高频事件
    • 避免闭包滥用、内存泄漏（监听、定时器、DOM 引用）

  2. 渲染优化（重中之重）

    • 减少重排重绘：集中操作 DOM、使用 transform/opacity
    • 虚拟列表 / 无限滚动处理大数据量
    • 避免强制同步布局（layout thrashing）

  3. 框架层面（Vue/React）

    • 合理使用 v-memo、computed、watch
    • 组件拆分、避免不必要渲染
    • 大数据列表必须用虚拟滚动

体验性能（体现你懂用户）

  • 骨架屏、加载状态、错误兜底
  • 交互优先：按钮、输入框等高优交互先响应
  • 滚动流畅、动画 60fps

监控与量化（高级工程师必备）

  • 接入性能指标：FP、FCP、LCP、CLS、TTI、FID
  • 线上实时监控：错误、卡顿、长任务、内存
  • 性能看板 + 告警 + 复盘机制
  • 用数据说话：优化前 vs 优化后

性能优化不是一次性工作，而是贯穿开发、构建、发布、线上监控的全流程体系。从业务出发，抓最影响用户的瓶颈，用最小成本拿到最大收益。

面试回答：
我做性能优化会从加载、运行、体验、监控四个方向系统性落地：加载上做构建优化、资源压缩、缓存策略、懒加载，提升首屏速度；运行上避免长任务、减少重排重绘、处理内存泄漏，框架层做精准更新和虚拟列表；体验上做骨架屏、交互优先，保证流畅度；最后通过 LCP、CLS 等核心指标做线上监控，形成可量化、可迭代的优化闭环。
