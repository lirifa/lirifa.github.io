什么是CSRF及基本防护手段

发表于 2022-04-08
攻击者诱导你在已登录的状态下，在目标网站上执行非你本意的操作。

什么是 CSRF

CSRF（Cross-Site Request Forgery）跨站请求伪造

简单理解：
攻击者诱导你在已登录的状态下，在目标网站上执行非你本意的操作。

核心一句话：
你已登录 A 网站 → 点开恶意网站 B → B 自动向 A 发请求 → A后端以为是你本人操作 → 攻击成功

攻击原理与流程

  1. 前提条件
    1. 用户已登录目标网站（Cookie 有效）
    2. 目标网站未对请求做身份校验（只认 Cookie，不认请求来源）
    3. 用户访问恶意页面，页面自动发起对目标站的请求
  2. 典型攻击流程
    1. 你正常登录 银行 / 论坛 / 后台系统，浏览器保存登录 Cookie
    2. 攻击者发给你一个恶意链接 / 图片 / 网页
    3. 你点开恶意页面
    4. 恶意页面自动构造请求（表单 / 图片 / 脚本）发送到目标网站
    5. 目标网站校验 Cookie 有效，认为是你本人操作，执行转账 / 删帖 / 改密码等操作

为什么能成功？关键原因

浏览器的 同源策略不阻止 “简单请求发送”，只限制读取响应。

  • 你访问恶意站 B
  • B 向 A 发请求
  • 浏览器会自动带上 A 域下的 Cookie
  • 服务器只看 Cookie，不校验请求是不是来自合法页面

这就是 CSRF 能成功的根本。

常见攻击载体

CSRF 不需要窃取 Cookie，只需要让浏览器自动带 Cookie 发请求。

  1. 自动提交表单（隐藏 form，页面加载自动 submit）
  2. 图片标签 <img src="https://bank.com/transfer?to=hacker&amount=10000">
  3. a 链接、iframe、Ajax 请求

只要能发起 HTTP 请求，就能做 CSRF。

危害

可以执行所有用户有权限但非本人意愿的操作：

  • 转账、充值、消费
  • 修改密码、绑定手机号
  • 发帖、删帖、关注、点赞
  • 管理员后台：添加用户、删数据、改配置

危害极大，尤其后台、支付、社交类系统。

CSRF 与 XSS 的区别

对比 CSRF XSS
全称 跨站请求伪造 跨站脚本攻击
核心 伪造用户请求 注入脚本执行
是否偷 Cookie 不需要 可以窃取 Cookie
信任关系 信任浏览器自动带的 Cookie 信任用户输入的内容
防御重点 校验请求来源、Token 转义输入输出、CSP
攻击条件 用户已登录目标网站（Cookie 有效） 只要能注入脚本即可

一句话区分：
XSS = 脚本在你页面里跑
CSRF = 借你的身份发请求

防御方案

1. 使用 CSRF Token（最通用、最推荐）

流程：

  1. 服务器在页面渲染时生成随机 Token，存入 Session
  2. 表单 / 请求里带上该 Token
  3. 后端校验 Token 是否一致
  4. 不一致直接拒绝

特点：

  • 恶意网站拿不到合法页面里的 Token
  • 无法构造有效请求
  • 前后端分离、传统 Web 都适用

2. 验证 Referer / Origin

  • Referer：标明请求来自哪个页面
  • Origin：标明请求来自哪个源（协议 + 域名 + 端口）

后端判断：必须来自本网站域名，否则拒绝

缺点：

  • Referer 可被伪造（弱防御）
  • 部分场景会被隐藏
  • 只能作为辅助手段

3. SameSite Cookie（现代浏览器强力防御）

给 Cookie 设置 SameSite 属性：Set-Cookie: sessionid=xxx; SameSite=Lax;
三种模式：

  • Strict：完全禁止跨站带 Cookie
  • Lax：仅安全跳转（a 链接）可带，POST / 异步请求不带
  • None：不限制（需配合 Secure）
    现代浏览器默认 Lax，大幅降低 CSRF 风险。

4. 敏感操作二次验证

  • 转账、改密要求输入密码 / 验证码
  • 即使被 CSRF，也无法完成攻击

5. 自定义请求头 + 后端校验

比如要求请求必须带：X-Requested-With: XMLHttpRequest
跨站无法随意添加自定义头（简单请求限制），后端可校验。

最佳实践

  1. Cookie 设置 SameSite=Lax + Secure
  2. 关键接口强制 CSRF Token
  3. 敏感操作验证码 / 二次密码
  4. 后端校验 Referer/Origin 作为辅助
  5. 避免使用 GET 请求执行修改操作（非常容易被 CSRF）

简单请求

简单请求是浏览器定义的、满足特定条件的跨域请求，同源策略允许其直接发送，但仍受 CORS 响应头控制；非简单请求会触发OPTIONS 预检，浏览器先探路再决定是否发送真实请求。
一个跨域请求必须同时满足以下所有条件，才会被浏览器判定为简单请求：

  1. 请求方法：只能是 GET、HEAD、POST 三者之一
  2. 请求头限制：仅允许包含以下 “CORS 安全头”，不能自定义其他头（如 Authorization、X-Token）：
    • Accept
    • Accept-Language
    • Content-Language
    • Content-Type（值必须是以下三种之一）
    • Content-Type 严格限定：
  3. Content-Type 严格限定：
    • application/x-www-form-urlencoded（表单默认）
    • multipart/form-data（文件上传）
    • text/plain（纯文本）
