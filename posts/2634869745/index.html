webpack基本概念

发表于 2021-06-22

Webpack 是一个现代 JavaScript 应用程序的静态模块打包器（module bundler）。在前端开发中，Webpack 用于将各种资源（如 JavaScript、CSS、图片等）打包成浏览器可以直接运行的静态资源包。

构建工具

早期开发网页，可以说是”刀耕火种“，都是一个个独立的文件，需要手动引入和管理。没有所谓的“模块化”的概念。
从2009年 nodejs 的出现，前端开发开始关注模块化和代码组织。
然后前端才开始出现工程化的转变，开始使用构建工具来管理项目。

早起构建工具

  • Grunt
  • Gulp

Grunt

  • 定位： 基于nodejs的任务运行器（task runner），用于自动化重复性任务。
  • 核心思想：配置优先（configuration over code）
  • 工作方式：
    • 写 Gruntfile.js 配置文件, 用JSON 风格描述任务
    • 每个任务：读文件 -> 处理文件 -> 写文件（三段式）
    • 频繁磁盘I/O 操作，导致构建速度慢。
  • 特点：
    • 配置繁琐，JSON多
    • 生态成熟、 插件多
    • 适合老项目、简单任务

Gulp

  • 定位：基于nodejs 的流式构建工具
  • 核心思想：代码优先（code over configuration）
  • 工作方式：
    • 写 gulpfile.js, 用JS 函数描述任务
    • 基于 Node Stream (流) + pipe() 管道
    • 边读边处理、边处理边输出，只读写一次磁盘
    • 速度块、内存友好
  • 特点：
    • api 极简、代码清晰
    • 流式处理，性能好
    • 现代前端项目常用

Grunt/Gulp 是任务工具，只管压缩、合并、编译这种简单流水线，它们不处理模块依赖关系。它们是无模块化时代的产物

Webpack 出现

Webpack是干嘛的？
定位：模块打包工具（module bundler）, 用于将多个模块打包成一个或多个文件，每个文件包含所有依赖的代码。
核心思想：万物皆模块（Everything is a module）
它解决的是 模块化问题：

  • 识别 import/ require
  • 构建依赖树
  • 打包、代码分割、懒加载
  • 处理图片、字体、CSS模块化
  • 兼容老浏览器
  • 热更新、开发服务器
    他是现代前端工程化的基石。

webpack基本构成

两个核心包跟一个核心插件：

  • webpack：核心包，负责模块打包
  • webpack-cli：命令行工具，用于执行webpack
  • webpack-dev-server：开发服务器，用于提供热更新、开发环境等功能

webpack基本配置

配置文件：webpack.config.js

  • mode：打包模式，可选值(production/development)，没有设置，默认为production
  • entry：入口文件，必填项，可以是字符串、数组、对象等
  • output：输出文件，必填项，指定打包后的文件输出目录和文件名等
  • plugins：插件
  • module：模块配置，用于处理不同类型的模块，如js、css、图片等
  • resolve：解析配置，用于解析模块路径，如@、~等
  • optimization：优化配置，从 webpack 4 开始，会根据你选择的 mode 来执行不同的优化， 不过所有的优化还是可以手动配置和重写。
  • devtool: 此选项控制是否生成，以及如何生成 source map。
  • devServer：开发服务器配置，用于提供热更新、开发环境等功能等
  • watch：监听文件变化，自动重新编译
  • stats：控制输出的详细程度
  • …

webpack的loader

loader的本质就是一个 「内容转换函数」，接收源码字符串，输出转换后的新字符串
loader 函数 return 出去的内容，必须满足三个条件：

  1. 必须是一个字符串
  2. 必须是 可执行的js 代码
  3. 必须是 模块格式
  • 职责：loader 是文件转换器，只针对单个文件内容做转换，不关心整体流程，只关心“进来什么 -> 出去什么”。
  • 工作时机：loader 在module 构建阶段运行，读取文件 -> 转译 -> 变成webpack能识别的js
  • 执行顺序：从右往左、从下往上链式执行

webpack的plugin

plugin的本质就是一个 带有apply方法的对象：

它做的事情：

  • 监听编译开始、结束、模块创建、代码生产、资源输出等所有阶段
  • 可以在这些阶段中插入自定义逻辑，实现自定义功能，如：修改配置、修改输出内容、删除文件、压缩代码、生成HTML、拷贝静态资源…。
  • 本质：扩展webpack能力，干预打包流程
  • 职责：流程拓展器，针对整个编译过程做事情，可以是打包前、打包中、打包后做任何事
  • 工作时机：在编译过程中，监听不同的事件，插入自定义逻辑，实现自定义功能
  • 执行顺序：按webpack钩子触发顺序执行。

Webpack 基于 ‌Tapable‌ 库提供丰富的生命周期钩子，主要分为两类：‌Compiler 钩子‌ 和 ‌Compilation 钩子‌。
常用 Compiler 钩子（全局构建阶段）：

  • compile：在新的编译开始前触发
  • emit: 最常用‌之一，在资源输出到输出目录（如 dist）‌之前‌触发
  • afterEmit‌: 在资源‌已输出到磁盘后‌触发，适合执行清理、通知等后续操作
  • done：最常用‌之一，在整个构建完成时触发，可用于统计、日志打印等
  • run：在编译开始前触发（仅在 webpack –watch 模式下有效）
  • failed：在编译失败时触发

常用 Compilation 钩子（单次构建阶段）:

  • ‌buildModule‌：模块开始构建前。
  • ‌succeedModule‌：模块构建成功后。
  • ‌optimizeDependencies‌：依赖优化阶段开始前。
  • optimize‌：整体优化步骤开始前。
  • ‌emitAssets‌：准备输出资源到输出目录前（与 emit 接近，但更早）‌
  • seal‌：编译密封（不可再修改）前触发。

Webpack 打包流程

  1. 初始化阶段

    1. 初始化参数(从命令行或者配置文件中获取参数)
    2. 创建compiler对象
    3. 初始化编译环境
    4. 开始编译 compiler.run()
    5. 确定入口文件

  2. 构建阶段

    1. 编译模块，从入口文件开始 loader 调用js解析器
    2. 完成模块编译，构建依赖关系图

  3. 封装阶段

    1. 合并 （根据依赖关系生成多个chunk）
    2. 优化 (TreeShaking(树摇), CodeSplitting(代码分割), Minification(压缩), scope hoisting(作用提升) )
    3. 输出资源 (将所有chunk合并到指定目录)
